# Журнал работы и хендовер для деплоя

Документ для продолжения работы с другого компьютера (где настроен SSH к ВМ).
Здесь — история решений этой сессии и точный план «что осталось сделать».

---

## Что сделано в этой сессии

### 1. SEO-поля для передачи новости на сайт
API сайта (`impactlaunch.ru/api/news`, см. `api-external.md`) поддерживает `meta_title`,
`meta_description`, `slug`, но генератор их не отдавал. Проработали по всей цепочке:

- **`generator.py`** — у площадки `zen` в схему статьи добавлены **обязательные** поля:
  - `meta_title` — SEO-заголовок 50–60 знаков, ключевой запрос в начале (может отличаться от
    видимого `title`: title продаёт клик, meta_title точно отвечает на поисковый запрос);
  - `meta_description` — сниппет выдачи 140–160 знаков, законченная фраза с ключами и пользой,
    не дубль meta_title;
  - `slug` — ЧПУ латиницей, 3–6 ключевых слов через дефис, транслит сути.
  - Инструкции добавлены в бренд-войс (`SYSTEM_BRAND`) и в чек-лист критика (пункт №8) —
    Sonnet доводит SEO-поля, а не оставляет черновыми.
- **`social_poster.py`** — `post_site` принимает и шлёт в API `meta_title`/`meta_description`/`slug`;
  добавлена `_clean_slug` (чистит ЧПУ до `[a-z0-9-]`, схлопывает дефисы, обрезает до 80 символов).
- **`bot.py`** — поля пробрасываются в `post_site` при публикации; в превью статьи перед
  публикацией показывается SEO-блок (под обязательный гейт — пользователь проверяет до отправки).
- `external_url` (дедупликация) пока НЕ используется: у бота нет URL первоисточника,
  новости пишутся из идеи. Если позже появится источник — стоит добавить.

### 2. Несколько пользователей бота
Раньше пускал только одного (`telegram_owner_id`). Добавлена поддержка списка.
- **`bot.py`** `is_allowed` теперь пускает владельца **и** всех из `telegram_allowed_ids`
  (если список пуст — пускает всех, поведение прежнее).
- **`config.json`** (НЕ в git): `telegram_allowed_ids: ["919942136", "838258851"]` — плюс
  владелец `416497736`.
- `config.example.json` — поле задокументировано.

### 3. Перенос на VPS (Яндекс.Облако, Docker)
Решение: **Docker-контейнер** с `restart: always`. Подготовлены:
- `Dockerfile` — `python:3.12-slim`, ставит зависимости, запускает `bot.py`. Секреты в образ
  НЕ запекаются.
- `docker-compose.yml` — `restart: always` (автозапуск после перезагрузки ВМ + рестарт при
  падении), `config.json` монтируется томом только на чтение, ротация логов (3×10 МБ).
- `.dockerignore` — исключает `config.json`, `.git`, доки, мусор.
- `DEPLOY.md` — пошаговый рунбук: SSH → Docker → перенос → запуск.

Ключевые решения по деплою:
- **Секреты вне образа.** `config.json` монтируется томом, не лежит в Docker-слоях и не в git.
  Менять конфиг можно без пересборки — достаточно `docker compose restart`.
- **Long-poll** → входящих портов не требуется, в Security Group ничего открывать не надо.
- Репозиторий уже на GitHub: `https://github.com/makar-nikshonkov/social-poster.git`.
  Деплой на ВМ — через `git clone`/`git pull`.

## ⚠️ Известная проблема: висячая запятая в config.json
В локальном `config.json` после `site_api_key` стоит запятая перед `}` — это **невалидный JSON**,
`python bot.py` упадёт на загрузке конфига. Перед запуском на ВМ убери запятую:
должно быть `...e683f3"` без запятой. (Файл не в git — правится только вручную на машине,
с которой переносишь.)

## Что осталось сделать (с компа, где настроен SSH)

Полные команды — в `DEPLOY.md`. Кратко:

1. **SSH-доступ к ВМ.** Добавить публичный ключ на виртуалку:
   - через `yc compute instance update --id <VM_ID> --metadata-from-file ssh-keys=...`
   - либо через серийную консоль в Console Яндекс.Облака.
   Нужны: `<VM_ID>`, `<VM_USER>` (пользователь ОС на ВМ), публичный IP `<VM_IP>`.
2. **Поставить Docker на ВМ:** `curl -fsSL https://get.docker.com | sudo sh`, затем
   `sudo usermod -aG docker $USER` и перелогиниться.
3. **Клонировать код:**
   `git clone https://github.com/makar-nikshonkov/social-poster.git social-poster`
4. **Перенести `config.json`** отдельным `scp` (его НЕТ в репозитории!) — и убрать висячую запятую.
   `telegram_allowed_ids` и все боевые токены уже внутри этого файла.
5. **Запустить:** `cd social-poster && docker compose up -d --build`,
   проверить `docker compose logs -f` — должно быть «Бот запущен: @ImpactLaunch_Posting_bot».
6. **Остановить локальный `python bot.py`**, если он где-то ещё крутится — Telegram отдаёт
   long-poll апдейт только одному получателю, два процесса будут конфликтовать.
